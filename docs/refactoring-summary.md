# Web-Printer 近期改动技术总结

## 概述

本次改动围绕**"高性能前端轮询 + 服务端缓存 + 代码质量"**三个维度展开，最终目标是让系统在高频交互下保持**"快、稳、省"**——前端不浪费请求，后端不重复计算，代码在未来易于理解和测试。

---

## 1. 客户端轮询的工业级实践

不只是 `setInterval`，而是一套完整的状态机：

| 机制 | 解决的问题 |
|------|-----------|
| **Page Visibility API** | 后台标签页降频/暂停，节省服务端资源 |
| **指数退避** (`2^failureCount`) | 服务端异常时自动降速，避免雪崩 |
| **抖动** (±15%) | 打散多客户端的请求峰值，避免惊群效应 |
| **请求去重** (`inFlight`) | 防止快速点击/定时重叠导致重复请求 |
| **动态频率** (5s busy / 15s idle) | 有活跃任务时实时追踪，空闲时减少开销 |

### 可迁移场景
任何需要轮询的后台监控、消息通知、进度追踪。

---

## 2. 短缓存 + 缓存失效 = 保护下游服务

前后端协同设计：

- 前端 5s/15s 轮询 → 后端 1.5s/3s 短缓存 → CUPS 实际调用大幅减少
- **写操作后立即失效**：打印/取消后 `invalidatePrinterState()`，确保下次读到最新状态
- **TTL 很短**（1.5s/3s）：牺牲极少实时性，换取大量性能提升

### 关键认知
缓存不是"越久越好"，而是"刚好够挡住重复请求"。

---

## 3. 内存管理：LRU + 空闲超时

`printerJobsCaches` 是 Map，有泄漏风险，解决思路：

1. **128 条目上限** + **LRU 淘汰**（按 `lastAccessAt` 排序，删最老的）
2. **10 分钟空闲超时**：打印机长期不被访问时自动清理
3. **inFlight 保护**：正在请求中的缓存条目不会被误删

### 可迁移场景
任何按 key 存储的临时缓存（会话、WebSocket 连接、请求去重）。

---

## 4. 事件循环中的定时器陷阱

`setInterval` vs 递归 `setTimeout`：

```
setInterval:  时间驱动  →  任务堆积  →  雪崩
setTimeout递归: 完成驱动 →  严格串行  →  安全
```

### 核心原因
Node.js 是单线程。如果清理任务耗时 15 分钟，`setInterval` 每 10 分钟塞一个任务进队列，导致无限追赶；`setTimeout` 递归是本轮完成后才开始计时。

### 可迁移场景
任何定时维护任务（日志清理、数据库归档、健康检查）。

---

## 5. 重构技巧：从"过程"到"意图"

server.js 的演变路径：

| 阶段 | 特点 |
|------|------|
| 原始 | 匿名箭头函数内联，缓存逻辑混杂，350+ 行 |
| 第一次 | 抽离 `lib/cache-manager.js`，按区域加注释分隔 |
| 第二次 | 匿名函数 → `handleXxx` 命名函数，路由注册集中 |
| 最终 | 启动流程拆成 4 个意图明确的步骤：`registerMiddleware` → `registerRoutes` → `ensureRuntimeDirectories` → `runStartupCleanup` → `scheduleMaintenance` |

### 学到的原则
- **一个函数只描述一个意图**（不是"做什么+怎么做"混在一起）
- **命名即文档**：`handlePrintDocument` 比 `async (req, res) => { ... }` 自解释
- **可测试性**：抽离后的 handler 可以直接单元测试，不需要启动 Express

## 6. 防御性编程细节

| 代码 | 防御了什么 |
|------|-----------|
| `removeFileIfPresent` 带空值守卫 | `request.file?.path` 为 undefined 时不抛异常 |
| `fs.rm(...).catch(() => {})` | 文件已被删除时的竞态 race condition |
| `cache.inFlight` 检查 | 淘汰/清理时跳过正在请求的缓存 |
| `hasActiveJobs` 竞态修复 | 被动切换打印机时立即 `loadJobs()`，避免状态滞后 |
